## Steam DRM stub on BadBloodGame.exe

- **Type:** SteamStub Variant 3.1 (x64)
- **Tool:** [Steamless](https://github.com/atom0s/Steamless) v3.1.0.5 by atom0s
- **Result:** Removed cleanly. Original ~1,154,680 bytes ‚Üí unpacked ~975,272 bytes.

Without the stub removed, the game fails to start outside of Steam:

| Steam state | Error |
|-------------|-------|
| Steam open  | `Application load error V:0000065432` |
| Steam closed | Steam launches and offers to install the game |
| ColdClientLoader | `Application load error 3:0000065432` |

Replace `BadBloodGame.exe` with the Steamless-unpacked version before
running anything else.

## Goldberg Steam Emulator

We use the **experimental** Goldberg build (the one that ships
`steamclient.dll` + `steamclient64.dll`). Without `steamclient64.dll`,
Steam intercepts the game launch when running.

Files dropped into the game folder:

- `steam_api64.dll` ‚Äî experimental Goldberg
- `steam_api.dll`   ‚Äî experimental Goldberg (32-bit, in case the game loads it)
- `steamclient64.dll`
- `steamclient.dll`
- `steam_appid.txt` containing `480` (Spacewar ‚Äî works around AppID 766370 being delisted)
- `steam_interfaces.txt` next to the DLL ‚Äî generated by Goldberg's helper
  tool against the **original** Techland `steam_api64.dll`
- `steam_settings/` folder, populated by the launcher per-account

Typical interfaces.txt produced for DLBB:
```
SteamClient017
SteamGameServer012
SteamUser019
SteamUtils009
SteamController006
```

The launcher writes these per-account files into `steam_settings/`:

| File | Content |
|------|---------|
| `force_account_name.txt` | username |
| `force_steamid.txt`      | session-derived SteamID |
| `custom_broadcasts.txt`  | matchmaking relay IP (your VPS, or 127.0.0.1 for local) |
| `disable_overlay.txt`    | empty file to disable Goldberg overlay |

Maintained Goldberg fork (since Mr_Goldberg's original repo went read-only):
<https://github.com/Detanup01/gbe_fork>

## Launcher gate in engine_x64_rwdi.dll

The engine refuses to run if it's not satisfied an "approved launcher"
is in charge ‚Äî this used to mean `BadBloodGameLauncher.exe`, but the
mechanism is actually EAC's `GameClient_Initialize` returning non-NULL.

When it returns NULL, the engine pops a MessageBox containing:
`Start the game using BadBloodGameLauncher.exe` and then exits.

### The check's code

```
0x006F73E9: B9 52 52 52 45            MOV ECX, 0x45525252       ; magic "ERRE"
0x006F73EE: 48 89 74 24 70            MOV [RSP+70h], RSI
0x006F73F3: FF 15 4F 52 0F 00         CALL [rip+0x0F524F]       ; launcher check
0x006F73F9: 48 8D 2D E8 44 1C 00      LEA RBP, [rip+0x1C44E8]   ; error string ptr
0x006F7400: 8B F0                     MOV ESI, EAX              ; save result
0x006F7402: 85 C0                     TEST EAX, EAX
0x006F7404: 74 73                     JE  +0x73                 ; ‚Üí 0x006F7479
```

### What the function actually does

1. `0x006F73A0` ‚Äî `LoadLibrary("EasyAntiCheat_x64.dll")`
2. `0x006F73B8` ‚Äî `GetProcAddress` of a factory (e.g. `CreateGameClient`)
3. `0x006F73CD` ‚Äî call the factory
4. `0x006F73CF` ‚Äî store the result in `[RDI+0x78]`
5. `0x006F73DE` ‚Äî `JNZ` to success at `0x006F74A2` if result non-NULL
6. Fall-through if NULL ‚Üí error MessageBox path

Both paths reachable from the `JE` at `0x006F7404` are error paths
(two different MessageBoxes). The actual clean exit point of the
function is `0x006F74E4`, reached via the `JNZ` at `0x006F73DE`.

### The patch

Replace the conditional `JNZ` with an unconditional `JMP` to the clean
exit:

| Address | Original | Patched |
|---------|----------|---------|
| 0x6F73DE | `0F 85 BE 00 00 00` (JNZ +0xBE) | `E9 01 01 00 00 90` (JMP +0x101 + NOP) |

This skips the whole EAC factory call and returns cleanly. The game
proceeds to the title screen.

### Things that didn't work

- NOP at `0x006F7404` ‚Üí fell into the error path
- JMP at `0x006F7404` ‚Üí jumped to a **different** error MessageBox

## EAC stub layout

### BEClient_x64.dll / BEServer_x64.dll
- 2 exports each: `Init`, `GetVer`
- Stub returns success / version string. Game never calls anything else.

### EasyAntiCheat_x64.dll
The real DLL exports 277 functions. Our stub exports the 50 that look
like actual EAC API surface:

- `Cerberus_*` (10 ‚Äî anti-cheat sample callbacks)
- `ClientAuth_*` (4)
- `CreateClientAuth`, `CreateGameClient`, `CreateGameLauncher`,
  `CreateHttpsClient`, `CreateThirdPartyLauncher`
- `GameClient_*` (10)
- `GameClientP2P_*` (15)
- `GameLauncher_*` (5)
- `NetProtectClient_*` (3)
- `ThirdPartyLauncher_*` (3)

The other ~227 exports are an embedded libconfig C++ library. None of
them have been observed being called by the game. If a future engine
build starts calling them we'd see "missing export" errors and would
need to add no-op stubs.

### EasyAntiCheat_x86.dll
The original DLL has zero named exports ‚Äî probably ordinal-only or
unused by the x64 game. The current stub is built as x64 because the
build environment lacks a 32-bit GCC. The x64 game loads the x64 stub
and never touches the x86 one; this only matters if you try to make
the 32-bit EAC launcher start.

## Server overview

- **Stack:** Python 3.10+ / FastAPI / SQLite
- **Default ports:** 80 (HTTP backend), 443 (HTTPS via stunnel), 47584 (Goldberg P2P relay)
- **Hosts redirect:** `127.0.0.1 pls.dlbb.com` (game still hits `pls.dlbb.com`; hosts file traps it)
- **TLS:** stunnel terminates 443, forwards plaintext to 80. Cert can be
  Let's Encrypt or self-signed ‚Äî game-side SSL verification is patched off.

### Endpoints implemented

All under one FastAPI app in `server/main.py`. See
[`server/README.md`](../server/README.md) for the full list with brief
descriptions.

### Catch-all behaviour

Any unmapped path returns `{"pls_result": 0, "pls_desc": ""}` (the
Techland "success" envelope) and logs the full request. Useful for
discovering new endpoints we haven't implemented ‚Äî launch the game,
watch the log, see which routes are firing.

## Open items

1. **Leaderboard index format** ‚Äî we have no Techland capture that includes
   played-account data, so the index payload format we serve might be
   missing fields the client expects. Why "Waiting for data" hangs.
2. **`playerdata.stats` array shape** ‚Äî same problem.
3. **Min-players = 6 hardcode** ‚Äî never reverse-engineered. Lives in
   `gamedll_x64_rwdi.dll`, not the engine.
4. **EAC x86 stub** ‚Äî built as x64, needs a 32-bit toolchain to rebuild
   correctly. Only matters if EAC launcher integration is wanted.
5. **libconfig exports** in the EAC stub ‚Äî not stubbed, may matter on
   newer engine builds.
